# 远程算力机加速拉取 GitHub 方案

> **场景**：协作者在中国境内的 A100 服务器上 `git clone` / `git pull` 我们的 GitHub 仓库速度太慢（通常 < 100 KB/s 甚至超时），影响每次更新代码的效率。
>
> **本文给出 5 种方案**，按"立即可用 → 长期最优"排序。任选一个或组合使用。

---

## 速选指南

| 你的情况 | 推荐方案 |
|---------|---------|
| 想立刻拿到代码（一次性） | **方案 A**（ghproxy 代理）|
| 想长期高速拉取（每天 git pull）| **方案 B**（Gitee 镜像）|
| 你本地拉 GitHub 很快，远程很慢 | **方案 C**（本地中转 scp）|
| 仓库太大或特别频繁拉 | **方案 D**（OSS 云盘中转）|
| 服务器有梯子或 socks 代理 | **方案 E**（proxychains）|

---

## 方案 A — `ghproxy` 反向代理（最快上手，30 秒）

把 GitHub URL 换成代理 URL，**不用任何配置**，直接跑 `git clone`。

### 一次性 clone

```bash
cd ~/project
git clone https://ghproxy.com/https://github.com/LIJianxuanLeo/hilserl-surrol-improved.git
git clone https://ghproxy.com/https://github.com/LIJianxuanLeo/hilserl-surrol-improved-v2.git
```

也可以用：
- `https://mirror.ghproxy.com/https://github.com/...`
- `https://gh.api.99988866.xyz/https://github.com/...`
- `https://kkgithub.com/...`（直接替换 github.com）

### 已经 clone 过的仓库改 remote

如果已经存在本地仓库（之前用原 GitHub URL clone 的），改 remote 即可：

```bash
cd ~/project/hilserl-surrol-improved
git remote set-url origin https://ghproxy.com/https://github.com/LIJianxuanLeo/hilserl-surrol-improved.git
git remote -v   # 验证
git pull        # 现在通过代理拉取
```

V2 同理。

### 优缺点

| ✅ 优点 | ⚠️ 缺点 |
|---------|---------|
| 30 秒上手，无需注册 | ghproxy 偶尔抽风（503） |
| 稳定速度 5-15 MB/s | 文件 > 100 MB 可能失败 |
| 命令完全兼容 git | 公益服务，可能被关 |

**推荐用法**：作为快速救场方案，先把代码拉下来再决定要不要做更彻底的方案 B。

## 方案 B — 镜像到 Gitee（强烈推荐，长期最优）

**Gitee** 是国内的代码托管平台，原生速度 50+ MB/s，且支持自动同步 GitHub 仓库。

### 一次性设置（5 分钟）

#### Step 1：协作者注册 Gitee 账号
访问 [gitee.com](https://gitee.com) 注册（手机号即可）。

#### Step 2：导入 GitHub 仓库
1. 登录后右上角 `+` → `从 GitHub/GitLab 导入仓库`
2. 输入 GitHub 仓库 URL：`https://github.com/LIJianxuanLeo/hilserl-surrol-improved`
3. 命名仓库（建议保持原名）
4. 选择 `公开` 或 `私有`
5. 点击 `导入` —— 后台自动同步，约 2-5 分钟

V2 同样操作一遍。

#### Step 3：（可选）开启自动同步
在 Gitee 仓库页面右上角 `管理` → `仓库镜像管理` → 添加镜像：
- 镜像方向：拉取
- 源 URL：`https://github.com/LIJianxuanLeo/hilserl-surrol-improved.git`
- 触发方式：定时（如每 30 分钟）

之后每当我在 GitHub push 新版本，Gitee 自动同步。**协作者从 Gitee 拉就行**。

### 远程服务器上的使用

```bash
# 首次 clone
cd ~/project
git clone https://gitee.com/<协作者用户名>/hilserl-surrol-improved.git
git clone https://gitee.com/<协作者用户名>/hilserl-surrol-improved-v2.git

# 已有仓库改 remote
cd ~/project/hilserl-surrol-improved
git remote set-url origin https://gitee.com/<协作者用户名>/hilserl-surrol-improved.git
git pull
```

### 优缺点

| ✅ 优点 | ⚠️ 缺点 |
|---------|---------|
| **国内速度极快**（50-100 MB/s） | 需要协作者在 Gitee 注册 |
| 永久稳定，无 503 | 同步有延迟（30 分钟级） |
| 100% git 命令兼容 | 私有仓库需付费（公开免费） |
| 可挂自动同步，零维护 | — |

**推荐用法**：长期使用首选。一次设置，终身受益。

## 方案 C — 本地中转：scp / rsync 推送

如果**协作者本地的 Mac/PC 拉 GitHub 速度可以**（比如有梯子或本地网络好），可以用本地做中转，避开远程服务器直连 GitHub。

### 一次性同步（首次部署）

```bash
# 在协作者的本地机器上
cd ~/Downloads
git clone https://github.com/LIJianxuanLeo/hilserl-surrol-improved.git
git clone https://github.com/LIJianxuanLeo/hilserl-surrol-improved-v2.git

# 打包（可选，rsync 会自动压缩传输，打包不必要）
# tar -czf v1.tar.gz hilserl-surrol-improved/

# 用 rsync 推到远程（推荐，支持断点续传）
rsync -avzP --exclude='.git/objects/pack/*.pack' \
    hilserl-surrol-improved/ \
    user@a100-server:~/project/hilserl-surrol-improved/

rsync -avzP --exclude='.git/objects/pack/*.pack' \
    hilserl-surrol-improved-v2/ \
    user@a100-server:~/project/hilserl-surrol-improved-v2/
```

> `-z` 启用压缩传输，`-P` 显示进度+断点续传。本地家庭宽带 100Mbps 上行约 12.5 MB/s，比远程拉 GitHub 通常快 100×。

### 增量更新（后续 `git pull` 替代）

每次主作者在 GitHub 推了新版本：

```bash
# 协作者本地：先把本地副本更新
cd ~/Downloads/hilserl-surrol-improved
git pull origin main

# 然后 rsync 推到远程（仅传输 diff）
rsync -avzP --delete \
    --exclude='outputs/' --exclude='.git/objects/pack/*.pack' \
    ./ user@a100-server:~/project/hilserl-surrol-improved/
```

`--delete` 让远程与本地完全一致；`--exclude='outputs/'` 防止覆盖远程的训练数据。

### 配置 SSH 别名简化命令

在协作者本地的 `~/.ssh/config` 加：

```
Host a100
    HostName <A100 服务器 IP>
    User <用户名>
    Port 22
    IdentityFile ~/.ssh/id_ed25519
    Compression yes
    ServerAliveInterval 60
```

之后命令简化为：
```bash
rsync -avzP ~/Downloads/hilserl-surrol-improved/ a100:~/project/hilserl-surrol-improved/
```

### 优缺点

| ✅ 优点 | ⚠️ 缺点 |
|---------|---------|
| 速度仅受本地上行带宽限制 | 需要协作者本地能拉 GitHub |
| 增量传输极快（只传 diff） | 多了一步本地操作 |
| 不依赖任何第三方服务 | — |

**推荐用法**：协作者本地有梯子时的首选。

## 方案 D — 云盘 OSS 中转（适合大文件 / 一次性传输）

打包成 tar.gz 上传到阿里云 OSS / 腾讯云 COS / 七牛云，远程服务器直接 wget。

### Step 1：本地打包

```bash
cd ~/Downloads
git clone https://github.com/LIJianxuanLeo/hilserl-surrol-improved.git
git clone https://github.com/LIJianxuanLeo/hilserl-surrol-improved-v2.git

# 打包（剔除 .git 历史可大幅减小体积）
tar --exclude='.git' -czf hilserl_repos.tar.gz \
    hilserl-surrol-improved/ \
    hilserl-surrol-improved-v2/

ls -lh hilserl_repos.tar.gz   # 通常 100-200 MB
```

### Step 2：上传到云盘

**阿里云 OSS**（推荐，国内速度快）：
```bash
# 安装 ossutil
brew install ossutil   # macOS
# 配置（一次性）
ossutil config

# 上传
ossutil cp hilserl_repos.tar.gz oss://<你的-bucket>/hilserl_repos.tar.gz

# 生成临时下载链接（24 小时有效）
ossutil sign oss://<你的-bucket>/hilserl_repos.tar.gz --timeout 86400
```

或直接用阿里云 / 腾讯云 / 百度网盘 等的网页上传。

### Step 3：远程服务器下载

```bash
ssh a100
cd ~/project
wget "<上一步生成的签名 URL>" -O hilserl_repos.tar.gz
tar -xzf hilserl_repos.tar.gz
ls   # 应看到 hilserl-surrol-improved/ 和 hilserl-surrol-improved-v2/
```

### 优缺点

| ✅ 优点 | ⚠️ 缺点 |
|---------|---------|
| 国内云盘速度极快（50+ MB/s） | 需要云存储账号 |
| 适合一次性大批量传输 | 不便于增量同步 |
| 可分发给多人 | 需手动管理签名链接过期 |

**推荐用法**：第一次部署、或需要把训练 checkpoint 传回时。

## 方案 E — 网络层代理（终极方案，需服务器有梯子）

如果远程 A100 服务器有 SOCKS5/HTTP 代理（公司或学校提供），可以让 git 全程走代理。

### Step 1：测试代理可用

```bash
ssh a100
# 假设代理地址是 127.0.0.1:7890（v2rayN/clash 默认）
curl -x socks5h://127.0.0.1:7890 https://github.com -I
# 期望看到 HTTP/2 200
```

### Step 2：让 git 永久走代理

```bash
# 全局设置 git 通过 socks5 代理
git config --global http.https://github.com.proxy socks5://127.0.0.1:7890
git config --global https.https://github.com.proxy socks5://127.0.0.1:7890

# 验证
git config --global --get-regexp http
```

之后所有 `git clone https://github.com/...` 自动走代理。

### Step 3：（可选）让 wget / curl 也走代理

在 `~/.bashrc` 加：
```bash
export http_proxy=socks5://127.0.0.1:7890
export https_proxy=socks5://127.0.0.1:7890
export no_proxy=localhost,127.0.0.1
```

`source ~/.bashrc` 生效。

### 优缺点

| ✅ 优点 | ⚠️ 缺点 |
|---------|---------|
| 一次配置，终身生效 | 需要服务器有可用代理 |
| 完全透明，任何工具都能用 | 部分公司/学校不允许 |
| 速度取决于代理带宽 | — |

**推荐用法**：服务器已有代理时的首选。

| 方案 | 上手时间 | 长期维护 | 速度 | 适用场景 |

| **A. ghproxy** | 30 秒 | 零 | 5-15 MB/s | 救急、临时使用 |
| **B. Gitee 镜像** | 5 分钟 | 零（自动同步） | 50-100 MB/s | **长期日常使用首选** |

## 相关：仓库本身大小检查

我们这两个仓库实际不大：

```
hilserl-surrol-improved/    主仓库 + 子模块
├── lerobot/                 ~50 MB（含演示数据 3.5 MB）
├── SurRoL_v2/               ~20 MB
├── docs/                    ~15 MB（含 PPT 8 MB）
└── .git/                    ~30 MB（git 历史）
合计                         ~115 MB
```

理论上即使 100 KB/s 也能在 20 分钟内拉完。如果当前速度更慢（< 50 KB/s），说明确实需要代理 / 镜像方案。

### 加速 git clone 的技巧（与上述方案叠加使用）

```bash
# 1. 浅克隆（只拉最新一次提交，省 90% 体积）
git clone --depth 1 <URL>

# 2. 单分支
git clone --depth 1 --single-branch --branch main <URL>

# 3. 不拉 LFS / 大文件
GIT_LFS_SKIP_SMUDGE=1 git clone <URL>

# 4. 不拉子模块（之后按需）
git clone <URL>   # 默认就不会自动拉子模块
# 子模块按需拉：
cd <repo>
git submodule update --init --recursive
```

对于咱们仓库，**`--depth 1` 浅克隆**可以从 ~115 MB 降到 ~85 MB，配合方案 A/B 可再快 25%。
